# Context Engineering 内存管理系统

Context Engineering 是 LLM 的内存管理系统，当 Prompt 承载力达到极限时，工程重心转向构建正确的上下文。

## 核心定位

**Prompt**："即时注入"
**Context Engineering**："内存管理"

## 上下文的六大要素

### 1. 系统指令（静态）
- 角色定义
- 行为约束
- 输出规范

### 2. 检索知识（RAG 拉取）
- 从外部知识库检索
- 动态加载相关内容
- 支持多源信息融合

### 3. 对话历史
- 涉及摘要技术
- 选择性遗忘机制
- 长对话管理

### 4. 工具定义
- 能力声明
- API 接口描述
- 使用规范

### 5. 结构化状态
- 任务进度跟踪
- 上下文状态维护
- 中间结果存储

### 6. 动态示例模板
- 按需加载示例
- 上下文相关模板
- 自适应提示

## 知识形态的演变

### 从静态知识图谱到动态本体

**动态本体定义**：
- 从静态 Schema 定义转向随数据演化的动态结构
- 配套需要 LLM 驱动或多 Agent 协作的本体维护

**核心特性**：
- 新概念自动纳入
- 旧概念自动退化
- 针对不同任务切分不同的知识"侧面"
- 支持过滤、筛选

## RAG 技术体系

### RAG 的完整定义

RAG 不是简单的"检索+生成"，而是一套**完整技术体系**：

#### 索引策略
- 不仅是检索，还需定义索引粒度
- 文档分块策略
- 多层级索引

#### 检索策略
- 关键词检索
- 向量化检索
- 混合检索
- 重排规则 (Reranking)

#### 知识冲突处理
- 建立多源信息矛盾时的优先级体系
- 冲突检测机制
- 信息可信度评估

## GraphRAG 架构

### 三层架构

1. **知识图谱层**
   - 结构化知识表示
   - 实体关系网络

2. **向量检索层**
   - 语义相似度计算
   - 快速相似内容查找

3. **LLM 推理层**
   - 深度理解与生成
   - 逻辑推理

### 核心技术

- **实体消歧**：区分同名不同义的实体
- **关系推理模式**：基于关系图谱的推理
- **社区检测规则**：发现知识簇和关联群体
- **深度结构化推理**：结合图结构和语义理解

## 核心挑战与解决方案

### 1. Token 管理

**问题**：多Token 不等于更好理解，过量信息反而降低模型性能

**解决方案**：
- 精确控制上下文长度
- 优先级排序
- 信息压缩

### 2. 渐进式披露 (Progressive Disclosure)

**策略**："先给地图、后按需深入"

**实现**：
- 逐步动态加载呈现
- 减少Token消耗
- 按需提供详细信息

### 3. 治理上下文污染 (Context Rot)

**问题**：长对话中出现的知识退化与噪声积累

**解决方案**：
- 定期上下文清理
- 信息质量评估
- 噪声过滤机制
